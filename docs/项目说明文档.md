# Simple RAG 项目说明文档

## 1. 项目概览

本项目是一个基于 **FastAPI + LangChain + LangGraph** 的 **Agentic RAG** 后端系统，面向真实知识库场景提供文档接入、混合检索、反思式回答生成与流式对话能力。系统同时集成了一套基于 **真实 RAG 执行** 的评测链路，用于离线评估检索与回答质量。

当前系统包含两条主线：

1. 在线服务主链路：文档导入、分层切块、向量与稀疏索引、混合检索、重排、LangGraph 对话生成、SSE 流式输出。
2. 离线评测主链路：数据集构建、按需人工审核、真实 RAG 执行、RAGAS 与检索指标评分。

系统当前强调两点：

1. 在线问答必须尽量走真实检索与真实 Graph，而不是 prompt-only 拼装。
2. 离线评测必须基于真实运行结果，而不是基于 AI 合成四元组自评。

## 2. 核心能力

### 2.1 在线 RAG 能力

1. 多格式文档接入：支持本地文件和 URL 文档导入。
2. 分层切块：使用父块与子块两级结构，兼顾召回粒度与回答上下文完整性。
3. 混合检索：同时使用向量检索和 BM25 稀疏检索，通过 RRF 融合提升召回稳定性。
4. 轻量重排：支持 Qwen reranker 对融合后的候选结果进行二次排序。
5. Agentic 对话：基于 LangGraph 组织检索、生成、重写、幻觉检查、有效性检查。
6. SSE 流式返回：支持 token 流、进度事件、最终答案与参考文档信息同步输出。

### 2.2 离线评测能力

1. 支持 `replay`、`synthetic`、`seed import` 三种数据集入口。
2. 支持将“数据集构建”和“真实测评执行”完全解耦。
3. 支持真实运行 `Graph + Retriever` 后再执行 RAGAS 和检索指标评分。
4. 支持人工审核表导出与审核结果回填。
5. 支持样本级记录、实验级汇总和 Markdown 报告输出。

## 3. 系统启动与运行流程

`app/main.py` 通过 FastAPI `lifespan` 统一管理系统初始化与关闭，启动顺序如下：

1. 加载 `config.yml` 与环境变量覆盖配置。
2. 初始化数据库连接与表结构。
3. 初始化 Embedding 模型。
4. 初始化向量库与父文档存储。
5. 初始化文档加载链、切块器注册表。
6. 初始化 Enhanced Parent Document Retriever。
7. 初始化重排器、Elasticsearch 与 Hybrid Retriever。
8. 构建并编译 LangGraph。
9. 注册 `document`、`retriever`、`conversation` 三组 API 路由。

关闭阶段则会显式执行：

1. `Graph.close()`，释放 checkpointer 与连接池。
2. `DatabaseManager.close()`，关闭异步与同步数据库引擎。

这使得在线检索、会话状态持久化和评测复用的运行时保持一致。

## 5. `app/core` 实现说明

### 5.1 `document_loader.py`

该模块负责统一处理多种文档来源，并在加载阶段补充统一元数据。

核心实现方式：

1. 定义 `DocumentLoader` 抽象基类，统一 `supports()`、`do_load()` 和元数据补全逻辑。
2. `_add_metadata()` 会为所有加载结果补充：
   - `path`
   - `file_directory`
   - `file_name`
   - `file_extension`
   - `last_modified`
   - `is_url`
   - `document_loader`
3. `DocumentLoaderChain` 以责任链方式注册并按顺序尝试各类 loader，直到命中可处理的实现。

当前具体 loader：

1. `PDFLoader`
   - 基于 `PyMuPDFLoader` 按页加载 PDF。
   - 为图片内容接入 `RapidOCRBlobParser`。
   - 支持表格抽取为 markdown。
2. `DoclingDocumentLoader`
   - 统一处理 `pdf/docx/xlsx/pptx` 等 Office 类文档。
   - 通过 `DocumentConverter` 做结构化转换。
   - 对复杂元数据进行过滤，减少后续持久化和序列化负担。
3. `TrafilaturaLoader`
   - 面向 URL 或 HTML 内容抽取主正文。
   - 适合网页类知识源的正文提取。
4. `HTMLLoader`
   - URL 走 `WebBaseLoader`，本地 HTML 走 `BSHTMLLoader`。
5. `TextLoader`
   - 直接读取文本、Markdown、代码文件。
   - 在 metadata 中记录 `text_length`，供后续是否启用 parent splitter 判断。

`load_doc_to_store()` 是文档导入主入口，主要职责：

1. 校验路径或 URL。
2. 调用 `DocumentLoaderChain` 加载原始文档。
3. 给文档打上 `file_id`。
4. 根据文件类型和文本长度决定是否启用父块切分。
5. 调用 `EnhancedParentDocumentRetrieverFactory.add_documents(...)` 完成切块与索引。
6. 将子块批量写入 Elasticsearch，供稀疏检索使用。

### 5.2 `chunking.py`

该模块负责根据内容类型选择合适的父块与子块切分器。

核心实现方式：

1. `SplitterRegistry` 在启动时注册父级 splitter 和 child splitter。
2. 父级 splitter 以“按格式优先”的方式路由：Markdown、HTML、代码、通用文本。
3. 子级 splitter 使用统一的较小 chunk 配置，用于生成子块索引。

关键实现点：

1. `MarkdownTextSplitterAdapter`
   - 先基于标题结构切分。
   - 通过 `merge_splits_by_structure()` 合并过小的相邻结构块。
   - 再交给递归切分器按长度切细。
2. `HTMLTextSplitterAdapter`
   - 与 Markdown 类似，但使用 HTML 标题结构。
3. `CodeTextSplitter`
   - 根据代码语言动态选择分隔符。
   - 兼顾语法边界与块大小控制。
4. `UniversalTextSplitter`
   - 作为兜底方案处理普通文本。
5. `TextSplitterMetadataMixin`
   - 会把实际经过的切分器名称写入 metadata，便于调试和回溯。

这套设计的意义在于：

1. 尽量保留 Markdown/HTML/代码的天然结构。
2. 父块更偏向语义完整性，子块更偏向检索粒度。
3. 对检索和回答两端都更友好。

### 5.3 `embeddings.py`

该模块通过 `EmbeddingModelFactory` 管理 embedding 模型单例。

实现方式：

1. 优先读取 `global_config.embedding`。
2. 按配置决定 embedding 来源：
   - OpenAI Embeddings
   - HuggingFace 远程推理
   - 本地 HuggingFace Embeddings
3. 本地模式下自动检测 CUDA，并按设备初始化模型。

这使项目可以在：

1. 纯本地 embedding。
2. OpenAI 兼容 embedding。
3. HuggingFace 远程推理。

三种模式间切换，而无需改动检索主逻辑。

### 5.4 `vector_store.py`

该模块管理向量存储与父文档存储。

实现方式：

1. `VectorStoreFactory` 统一创建并缓存：
   - `Chroma`
   - `LocalFileStore`
   - `KV docstore`
2. 向量库默认路径为 `store/chroma_langchain_db`。
3. 父文档存储默认路径为 `store/parent_docs`。

设计意义：

1. 子块向量检索走 Chroma。
2. 父块正文持久化走本地 docstore。
3. 检索时可以先召回子块，再回溯父块原文。

### 5.5 `retriever.py`

这是在线检索链路最核心的实现模块，包含 ES 稀疏检索、父文档检索、混合融合和预处理逻辑。

#### 5.5.1 Elasticsearch 稀疏检索

1. `ElasticSearchFactory` 负责初始化 ES client 和索引。
2. 索引中显式定义了 `page_content` 和常用 metadata 字段映射。
3. `EnhancedElasticsearchRetriever` 对 LangChain 原始 ES retriever 做了增强：
   - 自定义过滤查询。
   - 允许基于文件范围附加 metadata filter。
   - 自定义 document mapper，将 ES 返回结果还原为 LangChain `Document`。
4. 子块写入 ES 时 `_id` 使用文档自身 id，便于后续删除和回溯。

#### 5.5.2 Enhanced Parent Document Retriever

`EnhancedParentDocumentRetriever` 在 LangChain `ParentDocumentRetriever` 基础上扩展了完整的父子文档管理能力：

1. 在写入阶段显式生成和记录父块/子块 id。
2. 在 metadata 中补充：
   - `children_ids`
   - `children_count`
   - `parent_index`
   - `child_index`
3. 提供按子块回溯父块的同步和异步方法。
4. 提供按父块级联删除的方法。

这使得系统不仅能检索，还能把索引结构作为一等对象进行管理和调试。

#### 5.5.3 Hybrid Retriever

`HybridPDRetriever` 负责把稀疏检索、向量检索和父块回溯串起来。

主流程如下：

1. `reset_file_ids(...)`
   - 绑定当前会话或当前请求允许访问的文件范围。
   - 生成当前范围内的 BM25 检索器。
   - 计算当前检索参数。
2. `_prepare_retrieval(...)`
   - 基于当前 file scope 构造向量检索和 BM25 检索参数。
3. `invoke()` / `ainvoke()`
   - 先做 BM25 子块召回。
   - 再做向量子块召回。
   - 通过 RRF 做 rank fusion。
   - 回溯到父块。
   - 如启用 reranker，则对父块进行重排。
   - 最终截断到 `final_k`。

#### 5.5.4 检索预处理

`_NLPPreprocessor` 会对 query 做中英文混合预处理，主要包括：

1. NLTK 资源加载。
2. `jieba` 中文分词。
3. 英文 lemmatization。
4. 停用词与标点过滤。

这部分主要服务于 BM25 检索质量。

### 5.6 `reranker.py`

该模块负责对检索结果做二次压缩和排序。

当前实现包括：

1. `QwenNativeReranker`
   - 使用 `Qwen/Qwen3-Reranker-0.6B`。
   - 通过 yes/no 相关性打分 prompt 计算候选文档分数。
   - 取 top-n 返回。
   - 异步路径通过 `asyncio.to_thread()` 包装同步推理。
2. `SimpleCompressor`
   - 作为简单 fallback，只做截断。
3. `RerankerFactory`
   - 根据 `retriever.reranker.enabled` 控制是否启用真实 reranker。

### 5.7 `graph.py`

该模块定义了在线问答系统的 LangGraph 工作流，是系统最关键的编排层。

#### 5.7.1 Graph 运行时成员

`Graph` 类维护以下核心单例资源：

1. 编译后的 graph。
2. PostgreSQL checkpointer 连接池。
3. 响应模型与轻量模型。
4. 检索工具列表。
5. chat 配置。

#### 5.7.2 工具定义

当前 graph 内置两个工具：

1. `_retrieve(query, runtime)`
   - 调用 `HybridPDRetrieverFactory.get_instance().invoke(query)`。
   - 以工具消息形式返回序列化后的检索文档。
   - 同时把文档写入 graph state 的 `documents` 字段。
2. `_get_docs_page(file_id, offset, limit, runtime)`
   - 用于分页读取某个文件下的父块正文。
   - 主要服务于长文档追读场景。

#### 5.7.3 State 初始化与上下文压缩

1. `init_graph_state`
   - 重置 `documents`、`rewrite_count`、`generate_count`。
   - 用轻量模型结构化判断是否保留或更新 `original_question`。
2. `compact_tool_messages`
   - 将历史工具消息正文替换为占位符，避免上下文无限膨胀。
3. `summarize_conversation`
   - 当轮次超过阈值时，将旧会话压缩为摘要，仅保留最近轮次原文。

#### 5.7.4 决策与生成节点

1. `decide_retrieve_or_respond`
   - 注入系统提示词和会话摘要。
   - 绑定工具给响应模型。
   - 让模型决定直接回答还是调用检索工具。
2. `generate_answer`
   - 从最新 `ToolMessage` 中提取上下文。
   - 使用 `original_question` 与检索结果生成答案。
   - 生成后增加 `generate_count`。
3. `rewrite_question`
   - 当检索不相关或答案无效时，对问题进行重写。
   - 增加 `rewrite_count`。

#### 5.7.5 反思与控制节点

1. `handle_retrieve_result`
   - 判断检索结果是否相关。
   - 相关则进入 `generate_answer`。
   - 不相关则进入 `rewrite_question`。
   - 达到最大重写次数时强制进入回答阶段，避免死循环。
2. `check_hallucination`
   - 判断回答是否被检索上下文支持。
   - 若不支持则重新生成。
   - 达到最大生成次数则进入 usefulness 检查。
3. `check_usefulness`
   - 判断当前回答是否真正解决用户问题。
   - 有用则结束。
   - 无用则继续重写问题。

#### 5.7.6 Graph 主流程

当前工作流可概括为：

```text
START
  -> init_graph_state
  -> compact_tool_messages
  -> summarize_conversation
  -> decide_retrieve_or_respond
     -> 直接回答则 END
     -> 调工具则 retrieve
retrieve
  -> handle_retrieve_result
     -> generate_answer
     -> rewrite_question -> decide_retrieve_or_respond
generate_answer
  -> check_hallucination
     -> generate_answer
     -> check_usefulness
check_usefulness
  -> END
  -> rewrite_question -> decide_retrieve_or_respond
```

这是一条典型的“检索 -> 回答 -> 自检 -> 纠偏”闭环。

#### 5.7.7 持久化与可恢复能力

1. `Graph.build()` 会初始化 Postgres checkpointer，并将 graph 编译为可持久化状态机。
2. `conversation` 路由在流式输出过程中会记录 checkpoint id。
3. 如果中途异常，可基于 checkpoint 恢复对话状态。
4. `Graph.delete_thread(thread_id)` 可删除对应会话状态。

## 6. `app/routers` 实现说明

### 6.1 `document.py`

负责文档接入与文档树查看，主要提供：

1. 文档列表查询。
2. 父块、子块查询。
3. 本地文件导入。
4. URL 导入。
5. 文档删除。

### 6.2 `retriever.py`

负责检索测试与范围绑定，主要提供：

1. 当前引用文件范围的新增、查询、删除。
2. 面向检索调试的 query 接口。
3. 会话级检索范围控制。

### 6.3 `conversation.py`

负责真实对话与 SSE 流式输出，主要实现：

1. `/api/conversation/chat` 调用 `graph.stream(...)`。
2. `stream_mode` 同时订阅：
   - `messages`，用于 token 流
   - `custom`，用于进度事件
   - `updates`，用于捕捉工具结果和最终答案
3. 新建会话时自动生成 `conversation_id` 和标题。
4. 恢复会话时可使用 checkpoint。
5. 正常结束后落库用户消息、AI 回答和引用信息。
6. 出错时回写错误事件并保留 checkpoint，便于恢复。

## 7. 配置说明

### 7.1 当前关键配置

```yaml
env_override: false

database:
  url: postgresql+asyncpg://postgres:pg123456@localhost:5432/simple_rag

elasticsearch:
  url: https://localhost:9200
  username: elastic

chat_model:
  default: gpt-4o-mini
  light: gpt-4o-mini

embedding:
  model: Qwen/Qwen3-Embedding-0.6B
  openai:
    enabled: false
  huggingface_remote_inference:
    enabled: false

chunking:
  parent:
    chunk_size: 1000
    chunk_overlap: 120
  child:
    chunk_size: 256
    chunk_overlap: 50

vector_store:
  collection_name: default

retriever:
  final_k: 8
  reranker:
    enabled: true

chat:
  max_rewrite_time: 2
  max_generate_time: 3
  conversation_summarize_threshold: 10

text_file_length_threshold: 1500

debug:
  enabled: true
  docling_front: true
  trafilatura_front: true
  graph_visualization: false
```

### 7.2 配置重点

1. `database`：数据库连接与 checkpoint 持久化基础。
2. `chat_model`：默认回答模型与轻量控制模型。
3. `embedding`：向量化能力来源。
4. `chunking`：父块与子块大小直接影响检索与回答上下文质量。
5. `retriever.final_k`：最终返回给回答阶段的父块数量。
6. `chat.max_rewrite_time` / `chat.max_generate_time`：控制 graph 的反思回路上限。
7. `conversation_summarize_threshold`：决定何时压缩历史对话上下文。
8. `text_file_length_threshold`：短文本是否跳过父块切分的重要阈值。
