# Simple RAG 项目说明文档

## 一、项目概述

本项目是一个基于 **LangChain**、**LangGraph** 和 **FastAPI** 构建的 **智能检索增强生成（Agentic RAG）** 系统后端服务。核心功能包括：

1. **多模态文档接入**：支持本地文件（PDF、HTML、Markdown、代码文件等）的级联解析，并集成 Web URL 网页抓取读取。
2. **智能层次分块**：内置根据文件特征分发给适用的解析器（如 Markdown 标题切分、HTML 标题分块、编程语言切分适配）。并采用 Parent Document Retriever 模式建立级联索引。
3. **混合检索融合**：结合 BM25 稀疏关键词检索和 Embedding 密集向量语义检索，利用 RRF（Reciprocal Rank Fusion）统一排序融合。
4. **大语言模型二次重排**：支持无缝拔插基于 Qwen 等轻量级重排序模型（Reranker）的深度理解精排。
5. **反思式图流转回复**：融合最新的 **LangGraph Agent** 框架构建的状态机，不仅支持单纯的直接回复与文献增强检索，还深度嵌套了事实幻觉检测、回答效用评估与基于状态计数的内循环 **自我反思-检索重写 (Thinking-Rewriting)** 机制。同时内置实时状态持久化 Checkpoint 机制，并为最终的答案推送 **流式 (SSE) 终端 Token 打印** 交互响应。

**技术栈**：Python 3.12+ | FastAPI | SQLAlchemy (async) | psycopg (pool) | LangChain | LangGraph | ChromaDB | PostgreSQL | HuggingFace / OpenAI Embeddings | NLTK + jieba

## 三、各模块详细说明

### 3.1 应用入口 — `app/main.py`

FastAPI 应用入口，通过 `lifespan` 钩子处理所有资源的分配释放。**启动时按照依赖关系对各类单例基础设施进行装载并预热组件模型**：

```
global_config.load()
  → DatabaseManager.init() 
    → await DatabaseManager.init_db() (异步初始化数据库建表并校验)
      → EmbeddingModelFactory.init_embedding_model()
        → VectorStoreFactory.init_vector_store() (载入 ChromaDB 集合)
          → VectorStoreFactory.init_docstore() (载入 LocalFileStore KV存储)
            → DocumentLoaderChain.init()
              → SplitterRegistry.init()
                → EnhancedParentDocumentRetrieverFactory.init(vectorstore, docstore)
                  → RerankerFactory.init()
                    → HybridPDRetrieverFactory.init()
                      → Graph.build() (载入PostgresSaver checkpointer以及LLM并构建图)
```

关闭时触发对应模块的钩子回收（如 `Graph.close()` 回收 Postgres Pool，`DatabaseManager.close()`）。

### 3.3 核心逻辑引擎层 — `app/core/`

整体包含“检索预备库”（知识入库、解析切块）以及核心引擎（“图反思状态机工作流”）。

#### 3.3.1 `document_loader.py` — 文档加载（责任链模式）

- **`DocumentLoader`**（抽象基类）：定义 `supports(file_type, is_url)` 和 `do_load(path, file_type, is_url)` 接口，子类按 `order` 属性排序（越小优先级越高）
- **`PDFLoader`**：使用 `PyPDFLoader` + `RapidOCRBlobParser` 加载 PDF，支持 OCR
- **`HTMLLoader`**：URL 使用 `WebBaseLoader`，本地文件使用 `BSHTMLLoader`
- **`TextLoader`**：处理 `.txt`、`.md` 及各类代码文件，直接读取文件内容构造 Document
- **`DocumentLoaderChain`**：责任链管理器，`init()` 时注册所有 Loader，`do_load()` 时遍历找到第一个支持的 Loader 执行加载
- **`load_doc_to_vector_store()`**：顶层入口函数，完成 加载 → 分块 → 添加到 EnhancedPDRetriever 的全流程

#### 3.3.2 `chunking.py` — 文本分块（注册表模式）

- **`TextSplitterMetadataMixin`**：Mixin 类，为分块后的 Document 追加 `text_splitters` 元数据
- **分块器适配器**：
  - `MarkdownTextSplitterAdapter`：按 Markdown 标题层级分块
  - `HTMLTextSplitterAdapter`：按 HTML 标题层级分块
  - `CodeTextSplitter`：代码语言感知分块（`RecursiveCharacterTextSplitter` 子类）
  - `UniversalTextSplitter`：通用文本分块
  - `ChildTextSplitter`：子文档分块器（配合 Parent Document Retriever）
- **`SplitterRegistry`**（注册表）：
  - 管理文件类型到分块器的映射（`SplitterEntry`）
  - 支持 pipeline 模式：多个分块器按顺序处理同一文档
  - `split(docs, file_type)` 根据文件类型选择合适的分块策略

#### 3.3.3 `embeddings.py` — Embedding 模型工厂

**`EmbeddingModelFactory`**（单例），根据 `config.yml` 中 `embedding` 配置选择：
- OpenAI Embedding（`OpenAIEmbeddings`）
- HuggingFace 远程推理 API（`HuggingFaceInferenceAPIEmbeddings`）
- HuggingFace 本地模型（`HuggingFaceEmbeddings`，自动检测 CUDA）

#### 3.3.4 `vector_store.py` — 存储工厂

**`VectorStoreFactory`**（单例），管理两个持久化存储：

| 存储 | 实现 | 持久化路径 | 用途 |
|------|------|-----------|------|
| VectorStore | `Chroma` | `store/chroma_langchain_db/` | 子文档向量索引 |
| DocStore | `LocalFileStore` + `create_kv_docstore` | `store/parent_docs/` | 父文档键值存储 |

#### 3.3.5 `retriever.py` — 检索器（核心）

包含三大组件：

**① `EnhancedParentDocumentRetriever`**（`ParentDocumentRetriever` 子类）

在原版基础上增强：
- `add_documents` / `aadd_documents` 返回父文档 ID 列表
- 为父文档添加 `children_ids`、`children_count` 元数据
- 为父/子文档添加 `parent_index`、`child_index` 位置元数据
- 手动生成子文档 ID（在添加到 vectorstore 之前）
- 提供 `get_parent_docs()`、`get_child_docs()`、`delete_docs_cascade()` 等辅助方法
- 重写 `_get_relevant_documents` 直接返回子文档列表（用于 BM25 + 向量检索后统一回溯父文档）

**② `EnhancedParentDocumentRetrieverFactory`**（单例工厂）

负责创建和配置 `EnhancedParentDocumentRetriever` 实例，提供 `add_documents()` 方法自动根据文件类型选择 parent_splitter 并执行分块入库。

**③ `HybridPDRetriever`**（`EnsembleRetriever` 子类）

混合检索器，核心检索流程：

```
输入查询
  → BM25 关键词检索（子文档）
  → 向量语义检索（子文档）
  → RRF 融合（由 EnsembleRetriever 完成）
  → 截取 rrf_k 个子文档
  → 回溯对应的父文档（去重、保序）
  → 可选 Rerank（QwenNativeReranker）
  → 返回 final_k 个父文档
```

- 检索参数（`RetrievalParams`）根据子文档数量动态计算
- NLP 预处理（停用词过滤、词形还原、jieba 分词）提取到独立的 `_NLPPreprocessor` 类，避免 Pydantic `ModelPrivateAttr` 问题
- 提供同步 `invoke()` 和异步 `ainvoke()` 方法

**④ `HybridPDRetrieverFactory`**（单例工厂）

管理 `HybridPDRetriever` 单例的创建和初始化。

#### 3.3.6 `reranker.py` — 重排序器

- **`QwenNativeReranker`**（`BaseDocumentCompressor` 子类）：基于 `Qwen3-Reranker-0.6B` 模型，使用 Causal LM 的 `yes`/`no` 判断逻辑计算文档相关性评分，支持同步和异步（`asyncio.to_thread`）
- **`SimpleCompressor`**：简单截断，仅返回前 `top_n` 个文档（目前未被工厂使用）
- **`RerankerFactory`**（单例工厂）：仅在 `config.yml` 中 `retriever.reranker.enabled=true` 时创建 `QwenNativeReranker` 实例

#### 3.3.7 `graph.py` — LangGraph 智能代理图工作流

这是整个系统的核心推理引擎，基于 **LangGraph StateGraph** 构建了一套带自我反思能力的 Agentic RAG 对话流程。

**整体工作流程：**

```
START → summarize_conversation → decide_retrieve_or_respond
  ├─→ (有 tool_calls) → retrieve → grade_documents
  │       ├─→ (相关) → generate_answer → check_hallucination
  │       │               ├─→ (无幻觉) → check_usefulness_node → check_usefulness
  │       │               │                   ├─→ (已解决) → END
  │       │               │                   └─→ (未解决) → rewrite_question
  │       │               └─→ (有幻觉)  → generate_answer (重试)
  │       └─→ (不相关) → rewrite_question → decide_retrieve_or_respond
  └─→ (无 tool_calls, 直接回答) → END
```

**`Graph` 类（核心容器）：**

所有 Node 方法均为类方法（`@classmethod`），通过类变量 `_response_llm` / `_light_llm` 访问运行时资源。

| 类变量 | 类型 | 说明 |
|------|------|------|
| `_compiled_graph` | `CompiledStateGraph` | 编译后的 Graph 实例（单例缓存） |
| `_connection_pool` | `ConnectionPool` | psycopg 连接池（供 checkpointer 使用） |
| `_checkpointer` | `PostgresSaver` | LangGraph 状态持久化检查点（存入 PostgreSQL） |
| `_response_llm` | LLM | 主模型（默认 `gpt-5-mini`），用于问题决策和生成回答，绑定 `stream_answer` 标签 |
| `_light_llm` | LLM | 轻量模型（默认 `gpt-4o-mini`），用于结构化评分（相关性、幻觉、有用性）及标题生成 |
| `_chat_config` | dict | 从 `config.yml` 读取的 chat 参数 |

**各节点（Node）详细说明：**

| 节点 | 方法 | 说明 |
|------|------|------|
| `summarize_conversation` | `summarize_conversation` | 当对话消息数超过 `message_summarize_threshold` 时，调用 `_light_llm` 生成历史摘要，替换旧消息以控制上下文长度 |
| `decide_retrieve_or_respond` | `decide_retrieve_or_respond` | 调用 `_response_llm.bind_tools([retrieve])`，LLM 自主决定是否调用 retrieve tool；注入历史摘要作为系统消息；通过 `get_stream_writer()` 发送进度事件 |
| `retrieve` | `retrieve_node` | 自定义检索节点（替代 ToolNode）；从消息的 `tool_calls` 中提取 query，调用 `HybridPDRetriever.ainvoke()`；将 Document 列表存入 `state["documents"]`，序列化文本存入 `ToolMessage` |
| `generate_answer` | `generate_answer` | 基于 `state["documents"]` 和当前问题，调用 `_response_llm` 流式生成回答；通过计数控制最大重试次数（`max_generate_time`）|
| `rewrite_question` | `rewrite_question` | 使用 `_light_llm` 将原始问题改写为更适合检索的优化版本；通过计数控制最大重写次数（`max_rewrite_time`）；超限后改为直接回答 |
| `check_usefulness_node` | `_passthrough` | 透传节点，无实际逻辑，作为 `check_usefulness` 条件边的挂载点 |

**条件边（Conditional Edges）：**

| 来源节点 | 条件函数 | 可能的下一步 |
|------|------|------|
| `decide_retrieve_or_respond` | `tools_condition`（LangGraph 内置） | `retrieve` 或 `END` |
| `retrieve` | `grade_documents` | `generate_answer` 或 `rewrite_question` |
| `generate_answer` | `check_hallucination` | `check_usefulness_node` 或 `generate_answer`（重试） |
| `check_usefulness_node` | `check_usefulness` | `END` 或 `rewrite_question` |

**评估逻辑（Evaluator Nodes）：**

| 方法 | 使用模型 | 结构化输出 | 判断内容 |
|------|------|------|------|
| `grade_documents` | `_light_llm` | `GradeDocuments` | 检索到的文档是否与问题相关 |
| `check_hallucination` | `_light_llm` | `HallucinationCheck` | 生成的回答是否基于文档（无幻觉） |
| `check_usefulness` | `_light_llm` | `UsefulnessCheck` | 回答是否充分解决了用户问题 |

**Prompt Templates（内置提示词）：**

| 变量名 | 用途 |
|------|------|
| `GRADE_DOCUMENTS_PROMPT` | 文档相关性评分 |
| `REWRITE_QUESTION_PROMPT` | 问题改写优化 |
| `GENERATE_ANSWER_PROMPT` | 基于上下文生成回答 |
| `CHECK_HALLUCINATION_PROMPT` | 幻觉检测评分 |
| `CHECK_USEFULNESS_PROMPT` | 回答有用性评分 |
| `SUMMARIZE_CONVERSATION_PROMPT` | 历史对话摘要生成 |
| `DECIDE_SYSTEM_PROMPT` | 决策节点系统提示（绑定 retrieve tool 时注入） |
| `GENERATE_TITLE_PROMPT` | 对话标题自动生成 |

**关键辅助方法：**

| 方法 | 说明 |
|------|------|
| `build()` | 构建并编译 Graph，初始化 LLM 和 Checkpointer，注册所有节点和边 |
| `get_compiled_graph()` | 获取已编译的 Graph 实例 |
| `close()` | 关闭 psycopg 连接池，在 FastAPI lifespan shutdown 时调用 |
| `delete_thread(thread_id)` | 删除指定会话的所有 checkpoint 数据（对话删除时调用） |
| `generate_title_async(message)` | 调用 `_light_llm` 异步生成对话标题（新建对话时并发执行） |

**元数据过滤：**

在 `retrieve_node` 传给 LLM 的 `ToolMessage` 中，父文档元数据仅保留以下字段（`KEPT_METADATA_KEYS`），避免无关信息干扰 LLM：
- `file_id`：所属文件 ID
- `file_extension`：文件类型
- `parent_index`：父文档在原文件中的分块序号
- `Header 1`：Markdown/HTML 标题（如有）

**状态机配置参数（`config.yml` → `chat`）：**

| 参数 | 说明 |
|------|------|
| `max_rewrite_time` | 问题最大重写次数，超限后强制直接回答（默认 2） |
| `max_generate_time` | 回答最大重新生成次数，超限后直接输出（默认 3） |
| `message_summarize_threshold` | 触发摘要压缩的消息条数阈值（默认 10） |

## 四、设计模式汇总

| 模式 | 应用位置 | 说明 |
|------|---------|------|
| **工厂 + 单例** | `EmbeddingModelFactory`、`VectorStoreFactory`、`EnhancedPDRetrieverFactory`、`HybridPDRetrieverFactory`、`RerankerFactory` | `init()` 创建并缓存实例，`get_instance()` 获取 |

    chunk_overlap: 50

          │            SplitterRegistry.split() → [父文档块]         │

          │          ├─ ChildTextSplitter → [子文档块]               │
          │          ├─ 子文档 → ChromaDB (向量索引)                 │
          │          └─ 父文档 → LocalFileStore (KV存储)             │
          │                                    ↓                    │
          │        EmbeddedDocument → PostgreSQL (元信息)             │
          └─────────────────────────────────────────────────────────┘

          │                     检索流程                             │
          │                                                         │
          │  设置参考文档 → reset_file_ids()                        │
          │     ├─ 获取所有子文档 → 创建 BM25Retriever              │
          │     └─ 计算动态检索参数 (bm25_k, vector_k, rrf_k)       │
          │                                                         │
          │  查询 → HybridPDRetriever.ainvoke(query)                │
          │     ├─ BM25 关键词检索 → [子文档候选]                   │
          │     ├─ 向量语义检索   → [子文档候选]                    │

          │     ├─ 回溯父文档（去重、保序）                         │
          │     ├─ (可选) Reranker 重排序                           │
          │     └─ 返回 final_k 个父文档                            │
          └─────────────────────────────────────────────────────────┘

          ┌─────────────────────────────────────────────────────────┐
          │                  Agentic RAG 对话流程                    │
          │                                                         │
          │  POST /api/conversation/chat                            │
          │     ├─ 新对话: 生成 UUID → 并发调用 LLM 生成标题        │
          │     │         → 写入 conversation 表                    │
          │     └─ 续对话: 若有 checkpoint_id → 从异常点恢复         │
          │                                                         │
          │  LangGraph.stream() (stream_mode=["custom","updates",   │
          │                                   "messages"])          │
          │                                                         │

          │     └─ 消息数 ≥ threshold → LLM 生成摘要，压缩历史       │
          │                                                         │
          │  decide_retrieve_or_respond (LLM + retrieve tool)      │
          │     ├─ 有 tool_calls → retrieve_node                   │
          │     │     → HybridPDRetriever.ainvoke()                │
          │     │     → grade_documents (文档相关性评分)            │
          │     │         ├─ 相关 → generate_answer                │
          │     │         │     → check_hallucination (幻觉检测)   │
          │     │         │         ├─ 无幻觉 → check_usefulness   │
          │     │         │         │     ├─ 已解决 → END          │
          │     │         │         │     └─ 未解决 → rewrite      │
          │     │         │         └─ 有幻觉 → generate_answer    │
          │     │         └─ 不相关 → rewrite_question             │
          │     └─ 无 tool_calls → 直接回答 → END                  │
          │                                                         │
          │  SSE 推送:                                              │
          │     custom 事件 → 进度信息                               │

## 七、SSE 流式输出协议

`POST /api/conversation/chat` 接口返回 `text/event-stream` 格式的 SSE 响应，前端按 `event` 字段区分处理逻辑。

**SSE 数据帧格式：**

```
data: {JSON 字符串}\n\n
```

**各事件类型说明：**

| event 字段 | 对应类 | status | 说明 | 关键字段 |
|------|------|------|------|------|
| `Token streaming.` | `SseTokenVO` | `progress` | LLM 生成的单个 token，前端拼接显示 | `token` |
| `Answer generated.` | `SseAnswerVO` | `finished` | 最终完整回答（含全部 token），含参考文档和会话 ID | `answer`, `references`, `conversation_id` |
| `Error` | `SseErrorVO` | `error` | 异常信息（如未关联知识库、Graph 执行失败） | `message`, `code`, `conversation_id` |
| `Done` | `SseDoneVO` | `finished` | 流结束标志，前端收到后关闭连接 | — |
| 自定义进度事件 | dict | `progress` | Graph 各节点通过 `writer()` 发出的进度描述 | `event`（事件描述文本）, `status` |

**`SseAnswerVO.references` 结构示例：**

```json
[
  {
    "id": "文件UUID",
    "path": "/path/to/file.pdf",
    "file_name": "file.pdf",
    "parent_docs": [
      {
        "id": "父文档UUID",
        "parent_index": 2,
        "content": "父文档内容..."
      }
    ]
  }
]
```

**异常恢复机制：**

当 Graph 执行抛出 `CustomException`（如未设置参考文档）时：
1. 推送 `SseErrorVO` 给前端
2. 保存当前 `checkpoint_id` 到 `conversation.checkpoint_id`
3. 下次同一 `conversation_id` 发起请求时，从 checkpoint 恢复执行（无需重新输入消息）
